# 能力矩阵

本文件不再比较竞品，而是直接定义我方产品的能力结构：6 类业务 Agent 负责面向业务做事，统一平台负责把这些能力稳定、可追溯、可治理地跑起来。

## 矩阵标记说明
- `核心`: 该能力是该 Agent 的直接卖点，用户能明显感知
- `内建`: 该能力稳定存在于流程中，但不是第一卖点
- `依赖平台`: 该 Agent 需要这项能力，但由统一平台提供
- `底座`: 统一平台的共性能力

## 各 Agent 的能力重点

### 研究 Agent
- 强在“找信息、整理信息、带证据输出”。
- 适合展示市场扫描、可比项目整理、研究简报初稿。
- 权限与评测不是它单独实现，而是平台统一兜底。

### 交易 Agent
- 强在“跨步骤推进、跨角色协同、状态与阻塞项管理”。
- 适合展示交易管线、尽调清单、审批前材料汇总。
- 如果没有状态追踪、证据链和人工复核，它就会退化成普通摘要工具。

### 招商租赁 Agent
- 强在“线索到租约节点”的业务流程协同。
- 适合展示线索优先级、跟进动作、租约状态和风险提醒。
- 需要平台提供连接 CRM、租约台账和审批边界的能力。

### 估值资管 Agent
- 强在“参数、假设、情景、资产经营结果”的分析组织能力。
- 适合展示估值支持说明、情景对比、复盘摘要。
- 这类 Agent 不能脱离证据和人工确认，否则风险很高。

### 楼宇运营 Agent
- 强在“工单、异常、服务请求、运营事件”的过程协同。
- 适合展示异常识别、SLA 风险、升级路径和运营摘要。
- 它的核心不是聊天，而是任务闭环和跨团队协作。

### 文档报告 Agent
- 强在“把多源输入变成可复核结果物”。
- 适合展示报告初稿、引用清单、缺失字段、待确认事项。
- 它必须和证据追溯、人工复核一起出现，才有可信度。

## 为什么要有统一平台
如果只有 6 个业务 Agent，没有统一平台，就会出现三类问题：

- 每个 Agent 都要重复做状态管理、权限、日志和评测，无法规模化。
- 页面上只能展示结果，无法展示过程，售前很难讲清可信性。
- 后续进入 PoC 时，无法把不同 Agent 接到同一任务流和同一治理口径上。

## 售前讲法
- 先讲“业务 Agent 在做什么”，不要先讲技术名词。
- 再讲“为什么这些 Agent 能稳定工作”，用统一平台的 8 类能力兜底。
- 最后讲“如何证明可信”，回到证据、人工复核、权限审计和 Eval。
