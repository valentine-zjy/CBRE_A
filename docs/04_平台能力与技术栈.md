# 平台能力与技术栈

本文件只定义 L1 能力层，不定义具体框架或产品选型。这里的“技术栈”指支撑 6 类业务 Agent 的平台能力，而不是一张脱离业务的技术名词清单。

## 统一平台能力接口

`orchestration` `knowledge_rag` `doc_understanding` `tool_use` `multi_agent` `state_tracking` `governance` `eval_observability`

## 平台能力总表
| 平台能力 | 解决什么问题 | 支撑哪些业务 Agent | 售前里如何表达 | 后续实现里意味着什么 |
| --- | --- | --- | --- | --- |
| `orchestration` 任务编排 | 把多步骤任务变成可执行、可暂停、可重试的流程 | 全部，尤其交易、招商租赁、楼宇运营、文档报告 | 任务流、步骤卡、状态推进、下一步建议 | 需要流程定义、步骤状态、失败重试、人工插点 |
| `knowledge_rag` 知识检索 / RAG | 让 Agent 基于资料和证据回答，而不是凭空生成 | 研究、交易、招商租赁、估值资管、文档报告 | 证据卡、来源列表、可比项目、资料引用 | 需要知识接入、切片索引、检索策略、时效管理 |
| `doc_understanding` 文档理解与结构化抽取 | 从合同、报告、清单、表格中抽字段和关键条款 | 研究、交易、估值资管、文档报告 | 抽取字段表、条款摘要、缺失字段提醒 | 需要解析、OCR、字段 schema、校验与冲突提示 |
| `tool_use` 工具调用与系统连接 | 把 Agent 连到 CRM、租约、估值、工单、文档系统 | 全部 | “从系统读、经人工复核后回写”的边界说明 | 需要连接器、权限继承、读写控制、错误处理 |
| `multi_agent` 多 Agent 协作 | 把不同角色的 Agent 串成一个任务链 | 全部，尤其交易、研究、文档报告 | 角色分工、交接链路、协同看板 | 需要角色契约、中间产物协议、冲突仲裁 |
| `state_tracking` 状态管理与结果追踪 | 记录任务当前状态、版本和结果物 | 全部 | 时间线、任务状态、版本历史、结果物追踪 | 需要任务 ID、版本、artifact 存储、trace 记录 |
| `governance` 权限、审计与人工复核 | 保证 Agent 不越权、不黑箱、可回溯 | 全部 | 权限范围、提交复核、审计留痕、风险提示 | 需要权限模型、策略控制、审计日志、复核流 |
| `eval_observability` Eval 与可观测 | 判断结果质量是否可用，并能持续改进 | 全部 | KPI 面板、通过条件、异常告警、验收状态 | 需要评测集、评分规则、运行日志、告警与看板 |

## 分能力说明

### 1. `orchestration`
- 解决什么问题：把“研究一下”“推进一下”“整理一下”这类模糊需求拆成可见的任务流。
- 支撑哪些 Agent：交易 Agent、招商租赁 Agent、楼宇运营 Agent 最依赖；其他 Agent 也需要。
- 在售前里如何表达：展示从输入到结果物的步骤链，而不是只展示最终答案。
- 在后续实现里意味着什么：需要任务定义、步骤状态、失败处理、人工接管点。

### 2. `knowledge_rag`
- 解决什么问题：让 Agent 的回答和结论建立在资料检索与证据映射之上。
- 支撑哪些 Agent：研究 Agent 与文档报告 Agent最直接，交易和招商也会大量使用。
- 在售前里如何表达：展示“这条结论来自哪里”，而不是只说“模型会回答”。
- 在后续实现里意味着什么：需要资料接入、分层知识库、时效管理与引用规则。

### 3. `doc_understanding`
- 解决什么问题：把非结构化材料变成 Agent 能处理的字段和事实。
- 支撑哪些 Agent：文档报告 Agent、交易 Agent、估值资管 Agent、研究 Agent。
- 在售前里如何表达：展示字段抽取、条款摘要、冲突提示、缺失项提醒。
- 在后续实现里意味着什么：需要 OCR、解析器、字段映射、结构校验与异常处理。

### 4. `tool_use`
- 解决什么问题：让 Agent 不只会总结，还能读取系统状态并参与协同。
- 支撑哪些 Agent：全部；尤其招商租赁、楼宇运营、交易和估值资管。
- 在售前里如何表达：强调“连接系统，但不绕过权限与人工复核”。
- 在后续实现里意味着什么：需要连接器、API 协议、只读/回写边界与权限继承。

### 5. `multi_agent`
- 解决什么问题：让不同职能 Agent 在同一个任务里各司其职。
- 支撑哪些 Agent：研究 Agent 与文档报告 Agent、交易 Agent 与估值资管 Agent 的组合最典型。
- 在售前里如何表达：展示 orchestrator、analyst、reviewer 等角色之间的交接。
- 在后续实现里意味着什么：需要 Agent 契约、handoff 机制、冲突仲裁和共享上下文。

### 6. `state_tracking`
- 解决什么问题：让每个任务都能回答“做到哪一步了”“产出了什么”“谁确认过”。
- 支撑哪些 Agent：全部。
- 在售前里如何表达：展示任务状态、版本历史、结果物列表和负责人。
- 在后续实现里意味着什么：需要任务状态机、artifact 管理、trace 记录和恢复点。

### 7. `governance`
- 解决什么问题：让 Agent 输出始终处在受控、可审计、可复核的边界内。
- 支撑哪些 Agent：全部，而且是售前可信度的核心。
- 在售前里如何表达：展示权限范围、证据链、人工复核、不可自动执行的边界。
- 在后续实现里意味着什么：需要身份权限、策略引擎、审计日志、复核流和披露标记。

### 8. `eval_observability`
- 解决什么问题：判断 Agent 是否真的节省时间、减少漏项、保持可追溯。
- 支撑哪些 Agent：全部。
- 在售前里如何表达：展示 KPI、通过条件、失败类型与整改闭环。
- 在后续实现里意味着什么：需要标准 case、评测数据、质量告警、运行分析和验收门槛。

## 售前表达原则
- 先展示业务动作，再回挂到平台能力。
- 每项平台能力都要能指向至少一个业务 Agent 的实际任务。
- 不承诺未实现的框架或集成；只承诺已经定义清楚的能力边界。
