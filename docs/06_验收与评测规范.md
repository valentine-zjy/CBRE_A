# 验收与评测规范

本文件用于定义售前展示与后续 PoC 的可验证性标准。目标不是证明“模型很聪明”，而是证明这套 Agent 产品在商业地产任务里：

- 能节省时间
- 能减少漏项
- 能输出可追溯结果
- 能进入人工复核

## 适用范围
- 售前 Demo 验收
- PoC 场景验收
- 后续试点前的统一评测口径

## 核心验收原则
- 所有结果都只能是“建议态”或“初稿态”，不能越过业务审批。
- 所有结论都要能回溯到输入、证据或规则。
- 所有高风险输出都必须进入人工复核。
- 任何“待客户确认”信息都不能被包装成既成事实。

## 四个核心指标
| 指标 | 要回答的问题 | 演示口径 | PoC 口径 |
| --- | --- | --- | --- |
| 节省时间 | Agent 是否减少人工收集、整理、拼装时间？ | 用任务步骤和样例工时对比展示 | 用真实样本前后对比验证 |
| 减少漏项 | Agent 是否能显式识别缺失、阻塞、超时、待确认项？ | 展示缺失字段、阻塞项、风险提醒 | 用检查清单覆盖率和漏项率验证 |
| 输出可追溯 | 关键结论是否有证据、有来源、有版本？ | 展示证据链、引用卡、trace | 用证据覆盖率和引用完整度验证 |
| 可复核 | 输出是否能进入人工确认、驳回和接管流程？ | 展示复核入口、复核意见、状态变化 | 用通过率、驳回率、修改闭环验证 |

## 支撑指标
| 指标 | 作用 |
| --- | --- |
| 权限审计 | 证明 Agent 不越权、不黑箱 |
| 运行可靠性 | 证明任务失败后能重试、降级或人工接管 |
| Eval 与观测 | 证明结果质量可持续测量，而不是一次演示好看 |

## 六类 Agent 的最小验收对象
| Agent | 最小验证场景 | 必看结果物 | 必须人工复核的人 |
| --- | --- | --- | --- |
| 研究 Agent | 市场扫描与可比项目整理 | 市场摘要、可比项目清单、待确认项 | 研究负责人 |
| 交易 Agent | 管线与尽调协同 | 状态看板、阻塞项清单、下一步建议 | 交易负责人 |
| 招商租赁 Agent | 线索到租约节点推进 | 线索优先级、跟进计划、风险提醒 | 招商或租赁负责人 |
| 估值资管 Agent | 估值假设与情景对比 | 参数摘要、情景对比、复盘说明 | 估值或资管负责人 |
| 楼宇运营 Agent | 工单与异常事件协同 | 工单队列、异常告警、升级清单 | 运营或设施负责人 |
| 文档报告 Agent | 多源材料生成初稿 | 结构化初稿、引用清单、复核清单 | 报告签发责任人 |

## 统一评测样例模板
每个评测样例至少回答以下字段：

`case_id` `target_agent` `business_task` `input_pack` `expected_artifact` `must_have_evidence` `human_reviewer` `pass_criteria` `failure_modes[]`

## 首批评测样例
| case_id | target_agent | business_task | expected_artifact | must_have_evidence | human_reviewer | pass_criteria |
| --- | --- | --- | --- | --- | --- | --- |
| RESEARCH-001 | 研究 Agent | 汇总某区域市场动态并输出简报 | 市场摘要、可比项目清单 | 每条关键结论可回溯到来源 | 研究负责人 | 结论可追溯，待确认项显式标注 |

## 通过门槛

### 售前 Demo 通过条件
- 页面能在 3 分钟内讲清产品定位、6 类 Agent 和统一平台。
- 至少 3 条典型任务流能完整展示输入、动作日志、结果物和人工确认点。
- 所有关键输出都能打开证据链或复核入口。

### PoC 通过条件
- 至少 1 个优先场景达到“节省时间 + 减少漏项 + 可追溯 + 可复核”四项同时成立。
- 不出现高风险错误：无来源结论、越权输出、把待确认项写成事实。
- 业务方认可输出是“可进入下一步工作”的建议态结果，而不是仅供演示观看。

## 不可接受的失败类型
| 失败类型 | 说明 |
| --- | --- |
| 无来源结论 | 输出了关键判断，但无法回溯到证据 |
| 越权或越边界 | 输出建议绕过审批、签发或真实业务责任 |
| 漏掉关键阻塞项 | 关键节点、缺失字段、异常事件没有被识别 |
| 把假设写成事实 | `待客户确认` 的内容被写成确定结论 |
| 无法复核 | 输出没有复核入口、复核人或复核状态 |

## 验收结论建议格式
| 结论 | 含义 |
| --- | --- |
| `pass` | 达到展示或 PoC 目标，可进入下一阶段 |
| `conditional_pass` | 主体可用，但仍有边界或治理项待补齐 |
| `fail` | 结果不可信，不能进入下一阶段 |

## 与页面和 Demo 的关系
- 页面负责“把价值讲清”。
- Demo 负责“把任务流跑清”。
- 评测负责“把可信度讲明”。

如果三者不能闭环，售前展示就不能直接进入 PoC。
