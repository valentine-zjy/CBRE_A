# 演示页面规格

本文件把 [../05_页面信息架构.md](../05_页面信息架构.md) 落成 Demo 页面规格。页面目标不是展示一堆能力标签，而是让客户看到“业务任务如何被 Agent 执行”。

## 页面总规则
- 页面按 7 个区块组织，不再按“几屏技术页”组织。
- 每个区块都必须出现一个或多个 Agent 的明确动作。
- 页面先展示业务结果，再展示技术解释与治理约束。
- 所有输出均为 Demo mock，不触发真实系统动作。

## 1. 产品定位
- 页面目标：20 到 30 秒讲清产品定义和适用范围
- 对应 Agent：6 类业务 Agent 全部出现
- Agent 在这里做什么：以总览卡形式说明各自服务对象和代表结果物
- 用户要看到什么结果：一句话产品定义、适用客户、非目标边界、6 类 Agent 总览

## 2. 业务痛点
- 页面目标：让客户先看到自己的业务问题
- 对应 Agent：研究 Agent、交易 Agent、招商租赁 Agent、估值资管 Agent、楼宇运营 Agent、文档报告 Agent
- Agent 在这里做什么：把痛点翻译成“哪些任务适合交给哪类 Agent”
- 用户要看到什么结果：按业务线分组的痛点卡片，以及对应的 Agent 响应动作

## 3. 六类 Agent 全景
- 页面目标：把 6 类 Agent 的输入、动作、输出讲清楚
- 对应 Agent：全部
- Agent 在这里做什么：每张卡片展示 `输入 -> Agent 动作 -> 输出 -> 人工确认点`
- 用户要看到什么结果：6 张可快速扫读的 Agent 卡，能直接映射客户部门和岗位

## 4. 典型任务流
- 页面目标：证明 Agent 不是概念，而是真的会跑任务
- 对应 Agent：建议至少覆盖“研究 + 文档报告”“交易 + 估值资管 + 文档报告”“招商租赁 + 楼宇运营”
- Agent 在这里做什么：展示任务启动、动作日志、结果物生成、人工接管
- 用户要看到什么结果：输入包、动作时间线、结果物卡片、证据入口、复核入口

## 5. 平台能力
- 页面目标：把 Agent 背后的统一平台讲清楚
- 对应 Agent：全部，重点映射到当前任务流
- Agent 在这里做什么：回挂每一步动作依赖的 L1 平台能力
- 用户要看到什么结果：八类平台能力卡片，以及“哪类 Agent 在用它”的映射

## 6. 治理与可信
- 页面目标：回答客户“为什么我能相信这个结果”
- 对应 Agent：全部
- Agent 在这里做什么：展示证据链、人工复核、权限边界、失败处理
- 用户要看到什么结果：证据抽屉、复核状态、访问边界提示、评测摘要

## 7. 落地路线
- 页面目标：说明如何从演示进入 PoC
- 对应 Agent：交易 Agent、招商租赁 Agent、文档报告 Agent 为首批优先
- Agent 在这里做什么：展示首批场景、阶段目标和验收条件
- 用户要看到什么结果：`Demo -> PoC -> Pilot -> Scale` 路径，以及每一步的进入条件

## 页面必须展示的结果物
| 类型 | 说明 |
| --- | --- |
| Agent 输入包 | 任务上下文、资料来源、系统记录或模板 |
| Agent 动作日志 | 每一步做了什么、用了哪些平台能力 |
| 结果物 | 摘要、清单、草稿、建议、告警等 |
| 证据 | 来源、引用、版本、待确认项 |
| 人工确认点 | 谁来复核、何时接管、通过还是驳回 |

## 推荐交互方式
- 使用单页纵向结构，顶部提供区块导航。
- 典型任务流区块支持切换不同 Agent 组合。
- 技术解释层默认折叠，按需展开。
- 治理与可信区块必须可从任一关键结果物跳入。

## 页面验收标准
- 任一区块都能明确回答“Agent 在这里做什么”。
- 用户先看到业务任务和结果物，再看到治理和技术解释。
- 页面上看不到任何“自动执行真实业务动作”的按钮语义。
