# 技术解释层规格

本文件把 [../04_平台能力与技术栈.md](../04_平台能力与技术栈.md) 映射到演示页面。技术解释层只做一件事：解释 Agent 为什么能稳定、可追溯、可治理地工作。

## 使用原则
- 默认折叠，按需展开，不抢业务主叙事。
- 每一条技术解释都要对应一个页面区块和一个真实业务动作。
- 不展示具体框架名，不堆底层实现细节。

## 平台能力与页面映射
| 平台能力 | 对应页面区块 | 页面上怎么展示 | 用户看到什么 | 不该展示什么 |
| --- | --- | --- | --- | --- |
| `orchestration` | 典型任务流、落地路线 | 步骤时间线、任务状态、下一步动作 | Agent 不是一次回复，而是一条任务链 | 工作流引擎品牌或底层部署细节 |
| `knowledge_rag` | 六类 Agent 全景、典型任务流、治理与可信 | 来源卡、检索命中、引用映射 | 结论来自资料，不是无来源生成 | 向量库、索引参数等底层实现 |
| `doc_understanding` | 六类 Agent 全景、典型任务流 | 抽取字段表、缺失项、冲突提示 | 文档被转成结构化输入供 Agent 使用 | OCR 供应商或模型细节 |
| `tool_use` | 典型任务流、平台能力 | 连接器徽标、系统读写边界、回写前复核 | Agent 可以连系统，但不绕过权限 | 真实账号、真实接口地址 |
| `multi_agent` | 典型任务流 | 角色分工、交接链路、协同摘要 | 多个 Agent 分工协作，不是一个万能体 | 底层 prompt 编排细节 |
| `state_tracking` | 典型任务流、治理与可信 | 状态标签、版本、trace、artifact 列表 | 每个任务都能知道做到哪了 | 存储实现或表结构细节 |
| `governance` | 治理与可信 | 权限范围、复核节点、审计摘要、披露标记 | 结果不会越权、不会跳过人工 | 真实权限策略配置 |
| `eval_observability` | 治理与可信、落地路线 | KPI 卡、通过条件、异常告警、验收状态 | 这套 Agent 有统一评测，不是只看 demo 效果 | 内部报表系统或监控平台品牌 |

## 页面呈现顺序
1. 先展示业务结果物。
2. 再点开技术解释层，看到该结果物依赖哪些平台能力。
3. 最后在治理与可信区块看到这些能力如何被约束和验收。

## 披露规则
- 涉及内部系统、权限、审批链路的内容统一标记 `需客户确认` 或 `未公开`。
- 技术解释层只解释能力，不承诺已经接通客户真实系统。
