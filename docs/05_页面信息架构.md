# 页面信息架构

本文件定义售前页面的 7 个核心区块。页面不再按“技术屏数”组织，而是按“客户理解顺序”组织。

## 页面组织原则
- 先讲业务问题，再讲 Agent 如何处理，再讲平台如何支撑。
- 每个区块都必须能回答“我们的 Agent 在这里做什么”。
- 治理与可信必须出现，但不能早于业务价值。

## 1. 产品定位
- `section_name`: 产品定位
- `section_goal`: 用最短时间讲清“这是一套统一平台 + 6 类业务 Agent 的商业地产 Agent 产品”
- `mapped_agents[]`: [研究 Agent, 交易 Agent, 招商租赁 Agent, 估值资管 Agent, 楼宇运营 Agent, 文档报告 Agent]
- `user_message`: 这不是普通问答，而是能围绕商业地产任务交付结果物的 Agent 平台
- `proof_points[]`: 一句话定义、目标客户、6 类 Agent 概览、非目标边界
- `expected_takeaway`: 用户知道这是什么产品，知道不是单个聊天机器人

## 2. 业务痛点
- `section_name`: 业务痛点
- `section_goal`: 让客户先在自己的业务里看到问题，再进入 Agent 方案
- `mapped_agents[]`: [研究 Agent, 交易 Agent, 招商租赁 Agent, 估值资管 Agent, 楼宇运营 Agent, 文档报告 Agent]
- `user_message`: 商业地产团队真正痛的不是“不会问问题”，而是信息分散、流程断点、结果不可追溯
- `proof_points[]`: 研究资料分散、交易推进漏项、招商跟进断点、估值参数难复核、工单闭环慢、文档成稿慢
- `expected_takeaway`: 用户相信这是面向真实业务任务的产品，而不是 AI 口号

## 3. 六类 Agent 全景
- `section_name`: 六类 Agent 全景
- `section_goal`: 一次性讲清 6 类业务 Agent 的服务对象、动作和结果物
- `mapped_agents[]`: [研究 Agent, 交易 Agent, 招商租赁 Agent, 估值资管 Agent, 楼宇运营 Agent, 文档报告 Agent]
- `user_message`: 每一类 Agent 都对应一条明确的业务线，不做抽象堆词
- `proof_points[]`: 每张 Agent 卡片必须展示服务对象、输入、动作、输出、人工确认点
- `expected_takeaway`: 用户能把自己的团队映射到某一个或某几个 Agent 上

## 4. 典型任务流
- `section_name`: 典型任务流
- `section_goal`: 用可演示的任务流证明 Agent 真的会做事
- `mapped_agents[]`: [研究 Agent, 交易 Agent, 招商租赁 Agent, 估值资管 Agent, 楼宇运营 Agent, 文档报告 Agent]
- `user_message`: Agent 的价值体现在任务链路里，而不是体现在一句回答里
- `proof_points[]`: 至少展示三条代表性任务流，建议覆盖“研究 -> 文档”“交易 -> 估值 -> 文档”“招商 -> 运营”
- `expected_takeaway`: 用户看到输入、动作日志、结果物和人工接管点之间的完整链路

## 5. 平台能力
- `section_name`: 平台能力
- `section_goal`: 说明为什么这些 Agent 不是零散功能，而是共享统一底座
- `mapped_agents[]`: [研究 Agent, 交易 Agent, 招商租赁 Agent, 估值资管 Agent, 楼宇运营 Agent, 文档报告 Agent]
- `user_message`: 统一平台负责编排、检索、连接、治理和评测，让不同 Agent 可复用、可扩展
- `proof_points[]`: 八类 L1 能力映射、任务流与平台能力的对照关系、平台负责与 Agent 负责的边界
- `expected_takeaway`: 用户知道平台能力是为业务 Agent 服务，而不是单独出售的技术墙

## 6. 治理与可信
- `section_name`: 治理与可信
- `section_goal`: 回答“为什么我能相信这个 Agent”而不是“它能不能生成内容”
- `mapped_agents[]`: [研究 Agent, 交易 Agent, 招商租赁 Agent, 估值资管 Agent, 楼宇运营 Agent, 文档报告 Agent]
- `user_message`: 所有输出都要可追溯、可复核、可审计，并且不越过权限和人工边界
- `proof_points[]`: 证据链、人工复核、权限边界、异常处理、Eval 指标
- `expected_takeaway`: 用户相信这个产品适合进入 PoC，而不是只能做概念展示

## 7. 落地路线
- `section_name`: 落地路线
- `section_goal`: 说明从售前展示到 PoC、再到试点的收敛路径
- `mapped_agents[]`: [交易 Agent, 招商租赁 Agent, 文档报告 Agent]
- `user_message`: 第一阶段先验证高价值、低耦合、可评测的任务，再逐步扩展到更多 Agent
- `proof_points[]`: Demo 范围、PoC 优先场景、试点扩展条件、治理与验收门槛
- `expected_takeaway`: 用户知道如何从展示进入可落地验证，而不是一次性大而全

## 页面实施规则
- 任一页面区块都必须明确映射一个或多个 Agent。
- 任一区块都必须展示“Agent 在这里做什么”，不能只展示概念图。
- 平台能力区块必须能回挂到前面的业务任务流。
- 治理与可信区块必须出现证据、复核和权限边界。
